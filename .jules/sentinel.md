## 2024-06-25 - [Strict Domain Matching and Protocol Assignment Limitations]
**Vulnerability:** The domain validation used `.includes('evzone.com')`, which allowed SSRF/Open Redirects to subdomains of unrelated malicious domains (e.g., `evzone.com.attacker.com`). In addition, the `URL` API silently failed to overwrite unapproved protocols (e.g. `javascript:`) with `https:` when sanitizing, allowing an XSS vulnerability to slip through if the protocol validation was not strict.
**Learning:** Using `.includes` or loosely matching URLs to a domain is almost always vulnerable to SSRF and Open Redirect attacks. Furthermore, JavaScript's `URL` object doesn't throw an error when attempting to assign a standard scheme (like `https:`) to a non-standard protocol (like `javascript:`).
**Prevention:** Always use strict exact matching (`===`) or strict suffix matching (`.endsWith('.domain.com')`) for validation. Also, explicitly check and allowlist URL protocols (e.g., ensuring they are in `['https:', 'http:']`) before attempting to manipulate or rely on the parsed URL.
