## 2024-05-15 - [HIGH] Open Redirect Vulnerability via Unvalidated Dynamic `window.location.href` Assignments
**Vulnerability:** Multiple UI components (`src/features/auth/signed-out/Index.tsx`, `src/features/auth/continue/Index.tsx`, `src/features/apps/Apps.tsx`, `src/features/profile/components/ProfileSidebar.tsx`) dynamically assigned parameters (e.g., `redirect_uri`, `launchUrl`, `external` URL parameters) to `window.location.href` without validation, leading to potential Cross-Site Scripting (XSS) via `javascript:` schema injections or Open Redirects to untrusted external domains. Also, `src/features/auth/PatchOidcResume.tsx` suffered from a bypass due to using a weak string exclusion (`!segment.includes('/')`) instead of strong validation.
**Learning:** Checking for substrings like `!segment.includes('/')` to prevent path traversal or open redirects is insufficient because an attacker can URL encode slashes (e.g., `%2F`) or use backslashes (`\`) that the browser will normalize later. Dynamic variables assigned to JavaScript sinks (`window.location.href`, `location.assign`, `window.open`) must be fully validated against an approved protocol and domain allowlist, or verified against a strict regex if it's meant to be an internal ID/path.
**Prevention:** Always validate all user-controlled URLs before utilizing them in redirects or assignments. For the frontend React components in this codebase, utilize the `isValidUrl` exported from `src/sanitizers/url.ts` (which restricts protocols to HTTP/HTTPS and checks that the hostname is within the trusted `evzone.com` ecosystem) prior to modifying `window.location.href`. For dynamic internal path segments, use a safe alphanumeric allowlist regex like `/^[a-zA-Z0-9-_]+$/` to ensure it cannot be abused for path manipulation.
