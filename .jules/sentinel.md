## 2024-04-06 - Open Redirect and XSS in URL Sanitization
**Vulnerability:** The `isValidUrl` and `sanitizeUrl` functions in `src/sanitizers/url.ts` had two major issues. First, they validated domains using `.includes('evzone.com')` instead of exact matching or `.endsWith()`, which allowed Open Redirect bypasses (e.g., `evzone.com.attacker.com`). Second, `sanitizeUrl` attempted to override the URL protocol to `https:` but failed silently when given unapproved schemes like `javascript:`, allowing malicious URLs to pass through and create XSS vulnerabilities. Protocol-relative URLs (`//`) were also unhandled.
**Learning:** Checking domains with `.includes()` is dangerous because it allows bypasses by creating domains that contain the target string as a substring. Additionally, setting `URL.protocol` fails silently for special schemas (like `javascript:`), meaning sanitizers must explicitly validate against an allowlist and reject unapproved protocols instead of trying to mutate them.
**Prevention:** Use exact domain matching (`===`) or suffix checking (`.endsWith()`). Explicitly reject protocol-relative URLs (`//`). Explicitly define an allowlist of approved URL protocols (`http:`, `https:`) and fail closed (return an empty string) if a URL uses an unapproved protocol.
