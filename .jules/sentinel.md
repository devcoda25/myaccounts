## 2024-03-24 - Fix URL Sanitization and Domain Validation Bypass
**Vulnerability:** URL sanitization logic previously relied on `.includes()` to validate domains (e.g., `parsed.hostname.includes('evzone.com')`), which allowed attackers to bypass checks by registering domains like `evzone.com.attacker.com` or `attackerevzone.com`. Additionally, attempting to mutate `parsed.protocol = 'https:'` on a `javascript:` URL fails silently, leading to XSS vulnerabilities where `javascript:` URLs were not properly sanitized to `https:` or stripped.
**Learning:** The JS `URL` API behaves differently for special schemas (like `javascript:`). Changing `url.protocol` on these special schemas is silently ignored. Domain validation must not use partial substring matches (`.includes()`) as they trivially allow subdomains or completely different root domains to match the target string.
**Prevention:** Always use exact matching (`===`) or suffix matching (`.endsWith('.domain.com')`) for hostname validation. Instead of attempting to mutate the protocol of an unvalidated URL, explicitly check the initial protocol against a strict allowlist (e.g., `['http:', 'https:']`) and reject the URL completely if it does not match.
