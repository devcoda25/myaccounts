
## 2024-10-24 - Fix URL Sanitization and Open Redirect Validation Bypasses
**Vulnerability:** The application previously relied on loose string `.includes()` for validating domains in URLs (e.g., `url.includes('evzone.com')`), which is susceptible to domain bypasses like `evzone.com.attacker.com` causing SSRF/Open Redirects. Furthermore, the URL sanitization incorrectly tried to mutate the protocol property of a `new URL()` instead of using an allowlist, which fails silently for special schemas like `javascript:`, allowing XSS payloads to pass through un-sanitized.
**Learning:** URL properties like `protocol` may refuse updates depending on the original schema. The JavaScript `URL` API requires relative paths to be explicitly handled because `new URL('/path')` throws an exception, leading to legitimate relative paths being erroneously blocked or bypassed if fallback logic is poorly designed. Domain validation `.includes` checks are fundamentally flawed.
**Prevention:** Always use strict domain matching (`===` or `.endsWith()`) against a list of known trusted domains. When sanitizing URLs against XSS, explicitly allowlist accepted protocols (`http:`, `https:`, etc.) and reject others rather than attempting modification. Catch exceptions for `new URL()` and explicitly permit safe relative paths (`startsWith('/')` and prohibiting `//`, `./`, `../`).
