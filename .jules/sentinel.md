
## 2024-04-30 - Fix Open Redirect and XSS via weak URL validation
**Vulnerability:** URL validation (`isValidUrl`) used `.includes('evzone.com')` to verify domains, making it vulnerable to SSRF and Open Redirect via subdomains of an attacker's domain (e.g., `evzone.com.attacker.com`). Furthermore, URL sanitization (`sanitizeUrl`) attempted to enforce HTTPS by mutating `parsed.protocol = 'https:'`. However, the JavaScript `URL` API silently fails to overwrite certain special schemas like `javascript:`, allowing malicious URLs to slip through and enable Cross-Site Scripting (XSS).
**Learning:** Checking for safe domains using string `.includes()` is inherently unsafe; full exact checks (`===`) or strict suffix checks (`.endsWith()`) must be used. Additionally, when using `new URL()` to sanitize links, explicitly validating the protocol against a strict allowlist (e.g., `http:`, `https:`) is mandatory because mutating the protocol property can fail silently.
**Prevention:** Always use `.endsWith('.domain.com')` or `=== 'domain.com'` to validate trusted domains. Never rely on mutating `URL.protocol` to sanitize a URL string without explicitly rejecting unauthorized schemas (such as `javascript:` or `data:`) through an allowlist beforehand. When `new URL()` rejects safe relative paths, ensure strict string verification prevents path traversal and scheme-less absolute paths.
