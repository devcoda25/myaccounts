## 2024-05-03 - Fix URL Validation SSRF and Sanitization XSS Vulnerabilities
**Vulnerability:** The application was using weak domain validation (`parsed.hostname.includes('evzone.com')`) in `isValidUrl`, enabling SSRF/Open Redirect attacks via subdomains or lookalike domains (e.g., `evzone.com.attacker.com`). Additionally, `sanitizeUrl` attempted to mutate the protocol of special schemas (like `javascript:`) by reassigning `parsed.protocol = 'https:'`. Because the URL API fails silently when mutating `javascript:`, this allowed XSS via malicious payloads bypassing the intended HTTPS coercion. `sanitizeUrl` also failed to safely process relative paths due to the `new URL()` constructor rejecting them.
**Learning:** Checking URLs with string substring matches (`includes`) is inherently flawed for domain boundary validation. Furthermore, the `URL` API's immutable protocol behavior on special schemes like `javascript:` requires explicit protocol rejection via an allowlist, rather than attempting implicit coercion. `new URL()` throwing on relative URLs without a base causes functional regressions when sanitizing safe relative paths like `/foo`.
**Prevention:** Always validate domain strings using exact matches (`===`) or exact suffix matching (`endsWith('.' + domain)`). When sanitizing URLs against XSS, employ an explicit allowlist (e.g., `['http:', 'https:']`) and reject anything else instead of attempting to transform invalid protocols. Support relative paths as a fallback carefully by asserting safe pathing (`startsWith('/')` and avoiding `//`, `/\`, or `../`) to prevent functional breakages while maintaining security.
