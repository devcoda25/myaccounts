
## 2024-05-24 - Fix URL Validation Bypass and XSS in URL Sanitizers
**Vulnerability:** URL validation (`isValidUrl`) used a weak `.includes('evzone.com')` check, allowing an Open Redirect/SSRF bypass (e.g., `evzone.com.attacker.com`). Additionally, the `sanitizeUrl` function did not restrict schemas against a whitelist, permitting XSS attacks via `javascript:` and other schemas, and `new URL()` fails silently when parsing some malicious structures.
**Learning:** Checking subdomains with string matching functions like `.includes()` is dangerous because domain name structure goes right-to-left. A malicious Top-Level Domain (TLD) or domain name can include the trusted domain name as a substring. Also, the `new URL()` API does not inherently prevent `javascript:` schemas or throw errors for many malicious URLs.
**Prevention:** Always use exact matching (`===`) or strict Regex matching (e.g., `/^([a-z0-9-]+\.)*(evzone\.com)$/i`) to validate domain names. Enforce a strict protocol allowlist (e.g., `http:`, `https:`, `mailto:`, `tel:`) when sanitizing URLs to prevent XSS payloads. Handle safe relative URLs explicitly when using the `URL` constructor to avoid regressions.
