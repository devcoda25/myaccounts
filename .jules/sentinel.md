## 2024-05-24 - [CRITICAL] Prevent SSRF and XSS through weak URL sanitization
**Vulnerability:** The application was vulnerable to SSRF/Open Redirects because domain validation used `.includes('evzone.com')` (allowing `evzone.com.attacker.com`). It was also vulnerable to XSS because the URL sanitizer allowed unapproved protocols (like `javascript:`) by silently ignoring them when `URL.protocol` assignment failed, and it blocked valid, safe relative paths that threw errors during `new URL()` parsing.
**Learning:** Using `.includes()` for domain validation is highly insecure and trivially bypassed. Additionally, `new URL()` throws errors for relative paths, requiring explicit safe-fallback handling. Relying on `url.protocol = ...` to sanitize a URL fails silently for special schemas like `javascript:`.
**Prevention:** Always use strict exact matching (`===`) or suffix matching (`endsWith`) for domain validation. Always explicitly validate against an allowlist of protocols (`http:`, `https:`) and safely handle relative path fallbacks by preventing `//`, `/\\`, and path traversals (`../`).
