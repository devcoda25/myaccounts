## 2024-05-24 - Fix URL validation and sanitization bypasses
**Vulnerability:** The application was vulnerable to SSRF/Open Redirects because `isValidUrl` used `hostname.includes('evzone.com')`, which could be bypassed with domains like `evzone.com.attacker.com`. Additionally, `sanitizeUrl` attempted to enforce HTTPS by mutating `parsed.protocol !== 'https:'`, which silently fails for malicious protocols like `javascript:` due to the URL API's constraints. It also did not allow safe relative URLs like `/api/test`, breaking valid links.
**Learning:** Naive `.includes()` checks for hostnames are trivial to bypass for SSRF/Open Redirects. When using the JavaScript `URL` constructor, parsing malicious URLs like `javascript:alert(1)//domain.com` does not throw an error; it assigns a `javascript:` protocol and an empty hostname, circumventing naive hostname validation if the protocol is not first strictly checked against an allowlist. Modifying `url.protocol` fails silently for special schemas like `javascript:`.
**Prevention:** Always use strict regex or exact matching (`===`) for hostnames. Ensure sanitization functions explicitly validate against an allowlist (e.g., `http:`, `https:`) and reject unapproved protocols instead of attempting mutation. Explicitly allow safe relative paths (e.g., starting with `/` but not `//`, `./`, `../`) as fallbacks to prevent functional regressions while maintaining security against XSS.
